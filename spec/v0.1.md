# SYMIONE Execution Primitive — Specification v0.1

## What this is

A protocol specification for conditional payment execution. Any system may implement against this spec. This document describes the interface contract, not a particular codebase.

## Core concept

An **execution** is an atomic unit: agreement + funding + proof + validation + conditional payment release. Money moves only when conditions evaluate to true.

## The 4-call API surface (v1)

All routes below require an API key: `Authorization: Bearer <key>` or `X-API-Key: <key>`, except where noted.

### 1. Create execution

- **Method / path:** `POST /api/v1/executions`
- **Headers:** `Idempotency-Key` (required, unique per logical create)
- **Request (JSON):**
  - `title` (string, required)
  - `description` (string, required)
  - `amount` (decimal string/number, greater than zero)
  - `currency` (ISO 4217 lowercase, e.g. `usd`)
  - `proof_type` (`url` | `file`)
  - `validation_config` (object, optional) — deterministic and AI flags (see below)
  - `payer_email`, `payee_email`, `deadline_at` (optional)
- **Response:** `execution_id`, `status`, `next_action`, timestamps, optional `stripe_payment_intent_id`
- **Errors:** `400` bad input; `401` missing/invalid API key

### 2. Fund (Stripe client secret)

- **Method / path:** `POST /api/v1/executions/{execution_id}/fund`
- **Request:** `{ "return_url": "<string>" }`
- **Response:** `client_secret`, `payment_intent_id` — use Stripe.js / Elements to confirm; server-side webhooks transition execution to fundable/proof states.

### 3. Submit proof

- **Method / path:** `POST /api/v1/executions/{execution_id}/proof`
- **Query:** `async_validation` (boolean, default true)
- **Body:** URL proof `{ "url": "https://..." }` or file fields per implementation
- **Response:** execution snapshot
- **Errors:** `400` if agreement not in a state that accepts proof (e.g. not funded)

### 4. Get execution (poll)

- **Method / path:** `GET /api/v1/executions/{execution_id}`
- **Response:** current `status`, `next_action`, optional `confidence`

### Bootstrap (operator)

- **Method / path:** `POST /api/internal/api-keys`
- **Header:** `X-Admin-Token` matching server `ADMIN_BOOTSTRAP_TOKEN`
- **Response:** raw `api_key` (shown once)

### Webhooks (outbound)

- **Register:** `POST /api/v1/webhooks/endpoints` with `{ "url": "https://..." }`
- **Delivery:** `Symione-Signature` header; body includes `type` (v0.1 reference implementation: **`execution.updated`**), `data.execution_id`, `data.status`, `data.next_action`

## State machine

| Status | Meaning |
|--------|---------|
| `created` | Agreement exists; not yet advanced |
| `awaiting_funding` | Waiting for payment |
| `awaiting_proof` | Funded; waiting for proof |
| `validating` | Proof submitted; validators running |
| `manual_review` | Human review required |
| `failed` | Validation or business rules failed |
| `paid` | Payment captured / released per implementation |
| `cancelled` | Cancelled |

Transitions are driven by funding webhooks, proof submission, validation outcome, and capture/refund logic in the implementation.

## Proof types

- **url** — HTTP(S) endpoint as proof (reachable URL).
- **file** — Object storage reference (e.g. uploaded blob key + metadata).

## Validation `validation_config` (informal schema)

Deterministic (examples):

- `require_status_200` (boolean)
- `allowed_domains` (string[])
- `min_lighthouse_score` (number, implementation-defined measurement)

AI tiers (Anthropic — reference implementation):

- `use_ai_validation` (boolean) or `brief` / `ai_brief` (string) or `brief_match` (boolean)
- `quality_threshold` (number) — requests AI evaluation
- `validation_tier`: `"premium"` — use premium model tier (Sonnet) per implementation

Escalation:

- Server `VALIDATION_ESCALATION_THRESHOLD` (e.g. `0.75`) — Haiku confidence below this may escalate to Sonnet.

## Webhook events (contract vs reference)

**Contract (spec):** implementations SHOULD emit lifecycle events such as:

- `execution.created`, `execution.funded`, `execution.proof_submitted`, `execution.validation.started`, `execution.validation.completed`, `payment.released`, `payment.failed`, `execution.cancelled`

**Reference implementation (v0.1):** emits a single outbound type **`execution.updated`** with current `status` and `next_action` until granular types are added.

## SDK contract

A compliant SDK MUST:

- Send `Idempotency-Key` on create execution
- Send API key on all authenticated routes
- Expose create, fund, submit proof, get execution, and optional webhook registration
- Use HTTPS base URL from configuration

Minimum methods (conceptual): `createExecution`, `getExecution`, `fund`, `submitProof`.

## Versioning

This spec is versioned. Breaking changes increment the major version. **v0.1** — MVP primitive, single execution, no swarm orchestration.

## Limitations

AI-based validation is **probabilistic**. Implementations SHOULD combine deterministic checks, confidence thresholds, escalation, and manual review for high-stakes flows.
